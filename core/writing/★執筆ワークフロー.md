# ★執筆ワークフロー

このファイルは、HowMaマガジン記事制作の標準ワークフローを定義します。

## 執筆作業開始トリガー

`1_stock` にある構成案から着手するときは、人間がGoogle Driveにある受領データをダウンロードして `core/writing/1_stock/` に置いたあと、ユーザーが対象ファイルをメンションしたうえで「執筆作業開始」と明示した場合のみ、次の順で作業を始めます。

1. メンションされた対象構成案 `.md` が `core/writing/1_stock/` 配下にあることを確認する
2. `2_in_progress` に `yymmdd_title` 形式の記事フォルダを作る
3. `_templates/yymmdd_template` をその記事フォルダとして複製する
4. 元の構成案 `.md` を `1_stock` から新しい記事フォルダ内へ移動する
5. 以後の執筆、調査、進行管理はその記事フォルダ内で行う
6. 続けて、記事制作フローの 2「IDEで構成案の解釈と前提整理をする」から 5「CTAと図版予定位置を入れる」までを自動で進める

## 自動実行範囲

「執筆作業開始」トリガーで自動実行する範囲は、初期セットアップと記事制作フロー 2〜5 までとする。

- 1は人間が事前に行う
- 2〜5はAIが自動で行う
- 6以降は人間とAIの共同作業として進める
- 6の人間によるAIライティング精査が終わるまで、7以降には進まない
- 「執筆作業開始」トリガーを受けたAIは、記事制作フロー1を実行しない
- 1で受領データを所定場所に保存し、中身を確認して、ディレクター確認などで止める必要がないと人間が判断した場合のみ、「執筆作業開始」トリガーで2〜5を自動実行する

上記の初期セットアップは、原則として `scripts/writing/start_writing_article.ps1` を使って半自動化する。

実行例:

```powershell
powershell -NoProfile -ExecutionPolicy Bypass -File .\scripts\writing\start_writing_article.ps1 -SourcePath ".\core\writing\1_stock\構成案.md" -MoveSource
```

補足:

- 「執筆作業開始」トリガーで半自動化する場合は、`-MoveSource` を付けて `1_stock` から移動する
- 手動テストや確認目的で残したい場合のみ、`-MoveSource` を付けずにコピーする
- `-Title`、`-Date`、`-Step` を指定すると、記事フォルダ名と本文ファイル名を調整できる
- `SourcePath` は必ず `core/writing/1_stock/` 配下の構成案を指定する。別フォルダのファイルでは実行しない
- スクリプトはテンプレート複製、元原稿配置、本文ファイル名変更、`brief.md` / `log.md` の初期入力まで行う
- スクリプトで初期セットアップしたあと、エージェントは同じ作業ターン内で記事制作フロー 2〜5 を続けて実施する
- 記事制作フロー 6 以降は、人間とAIの共同作業として進める
- 記事制作フロー 7 以降は、6「人間によるAIライティングの精査作業」が終わってから再開する

## 記事制作フロー

記事制作は、原則として次の流れで進めます。

1. <mark style="background-color: #ffcccc;">【人間】</mark>Google Driveにある受領データをダウンロードし、構成案を通読する
   - 受領データは `core/writing/1_stock/` に置く
   - ディレクター指示、元原稿、補足資料、記事の目的を確認する
   - ディレクター確認が必要な点、素材不足、前提の不明点がある場合は、この時点で止める
   - そのまま進めて問題ないと判断した場合のみ、対象構成案をメンションして「執筆作業開始」と依頼する
   - 人間が気になった点や前提は、必要に応じて `brief.md` に残す
2. 【AI自動作業】IDEで構成案の解釈と前提整理をする
   - チェック結果は `brief.md` に記録する
   - 構成案から読み取れる「この記事の狙い」を端的な一覧にし、`outline.md` に `## 構成案から読み取れる記事の狙い` として残す
   - 記事の狙いには、読者の状況、不安、記事の主目的、渡す判断軸、読後のゴール、CTA接続、他ステップとの境界を含める
   - 2-1: 構成案の矛盾や整合性をチェックする。ここでは構成案内でのみ確認する
   - 2-2: ソース記事から、矛盾や論理破綻がないか整合性チェックをする
   - 2-3: 執筆記事に参照記事として使えそうな記事を検索し、該当しそうであればURLと記事タイトルを候補として記録する
   - 2-4: 構成案を考慮しつつ、記事の構成を再編する
   - 2-5: 構成案から変えた場所、追記した内容などを `brief.md` の作業ログに残す
   - 2-6: Researchとファクト確認として、数値、制度、表現、HowMaのこと、不動産業界としての定説などから問題ないかを確認する
   - 2-6で問題がある場合は、補足、追記、リサーチを行い、どうすべきか人間の判断を仰ぐ
3. 【AI自動作業】`brief.md` と構成案を受けて、`outline.md` へ記事の構成を出力する
   - 人間が確認できる状態にする
   - 見出し構成、記事全体の流れ、構成案から変更した意図が追えるようにする
4. 【AI自動作業】`post-<管理番号>_stepx-x_<短いタイトル>.md` にドラフトを出力する
   - `outline.md`、構成案、`brief.md` を元に本文を書く
   - 本文は、導入、本文、まとめのルールで執筆する
   - ソース記事や参照記事候補から、本文内に入れる関連記事のタイトルとリンクを選び、読者向け導線として `関連記事：` 表記で配置する
5. 【AI自動作業】CTAと図版予定位置を入れる
   - CTAは文字CTAとして配置する
   - CTA位置は `<mark style="background-color: #ffcccc;">CTA</mark>` の赤マーカーで示す
   - 本文用の図版場所を検討し、`図版予定` という文字を追加して赤マーカーを引く
   - ここまでが「執筆作業開始」トリガーで自動実行する範囲
6. <mark style="background-color: #ffcccc;">【人間】</mark>AIライティングの精査作業を行う
   - AIが書いた本文を人間が読み、違和感、過不足、言い回し、読者視点、記事の軸とのズレを確認する
   - 修正指示や判断メモがあれば `brief.md` または `log.md` に残す
   - この精査が終わるまで、7以降の作業には進まない
7. <mark style="background-color: #fff2cc;">【人間+AI】</mark>タイトルを見直す
   - 読者の検索意図、記事の約束、本文内容とのズレがないか確認する
8. <mark style="background-color: #fff2cc;">【人間+AI】</mark>H2の導入を修正する
   - H3など下層見出しの内容を加味して、各H2の導入文を調整する
9. <mark style="background-color: #fff2cc;">【人間+AI】</mark>図版を作る
   - 本文中の `図版予定` をもとに、本文用図版を作成する
   - AIがラフ案、文言、構成、素材案を作り、人間が意図、正確性、見せ方を確認する
   - 完成した素材は `assets/description/` に置く
10. <mark style="background-color: #fff2cc;">【人間+AI】</mark>アイキャッチを作る
   - 記事内容と訴求がずれないように整える
   - AIがラフ案、文言、構成、素材案を作り、人間が訴求、トンマナ、違和感を確認する
   - 完成した素材は `assets/thumbnail/` に置く
11. 【AI】校正・修正する
   - 矛盾、誤字脱字、てにをは、表記ゆれ、重複を確認する
   - 論理の流れ、トンマナ、読者視点のわかりやすさ、CTA接続を確認する
   - 同じ言い回し、同じ語尾、同じH2導入が続いていないか確認し、重複感がある表現は変更する
   - `〜を整理します`、`ここでは〜を解説します`、`〜について見ていきます` などの定型表現が多い場合は、過去の執筆記事に近い自然な表現へ置き換える
   - `core/writing_guide/tone_and_manner.md` の `Expression Fit` と `Repetition Control` を参照する
12. 【AI】ファクトチェックを最終チェックする
   - 執筆後に事実関係をもう一度確認する
   - 数値、制度、出典、HowMa機能、関連記事との整合性を確認する
13. <mark style="background-color: #ffcccc;">【人間】</mark>最終通読を行う
   - 記事全体を読者目線で通読し、違和感、説明不足、言い回し、CTA導線、納品状態を確認する
   - 必要な修正があれば `brief.md` または `log.md` に残し、AIへ修正依頼する

## 作業終了時の作業報告フロー

作業を途中で止めるとき、または自動作業範囲が終わったときは、必ず作業報告を出力する。

- ユーザーが「コラビット終了」と依頼した場合、この作業報告フローを起動する
- 「コラビット終了」はスクリプトではなく、AIがその時点の作業内容を確認して都度生成する
- 対象記事フォルダの `log.md` に、作業報告を追記する
- チャットの最終返答にも、`log.md` と同じ作業報告を出す
- 作業報告はテンプレートの空欄を残さず、AIが作業内容に合わせて中身まで自動生成する
- 作業報告には「どこまでやったか」と「次はどこから再開するか」を必ず書く
- ディレクター向けと自分向けの両方を生成する
- Notionへ作業報告を残す場合も、まずチャットに作業報告案を出力する
- ユーザーが作業報告案の内容を確認し、「OK」「Notionに追加して」などと明示した場合のみ、Notionのコラビット専用DBにページを追加して同じ内容を残す
- Notionへ直接書き込む前に、必ず人間確認を挟む

作業報告の形式は、`core/writing/_templates/作業報告テンプレート.md` を使う。Notionやディレクター向けには同ファイル内の「ディレクター向け」、再開用の詳細メモには「自分向け」を使う。

AIは作業報告を作る前に、対象記事フォルダの `log.md`、今回更新したファイル、現在のワークフロー工程を確認し、次の内容を推定して埋める。

- 作業時間: 分かる範囲で今回の作業時間を書く。不明な場合は `時間未記録` とする
- 進捗: 完了したStep、残っているStep、FB待ちなどの状態を書く
- 作業内容: 実際に更新した内容だけを書く
- 残作業: 次に必要な作業を具体的に書く
- 次回の開始位置: ワークフロー番号と作業名で書く
- 確認事項: 人間判断が必要な点だけを書く

Notion登録フロー:

1. AIが作業報告案をチャットに出す
2. ユーザーが内容を確認する
3. ユーザーがOKした場合のみ、AIがNotionのコラビット専用DBにページを追加する
4. Notionへ追加したら、追加先ページ名またはURLをチャットで報告する

ディレクター向けは、原則として次の形式で出す。

```md
▼作業報告　yyyy mm/dd｜ HH:MM～HH:MM

【進捗】
・StepX-X　完了（FB待ち / 残：FB対応・WP作成・ジェネレート）
・StepX-X　残：◯割

【作業内容】
・StepX-X　構成案確認、ドラフト作成
```

自分向けには、作業対象、今回の到達点、完了した工程、更新したファイル、判断・変更メモ、未対応、次回の開始位置、人間に確認してほしいことを残す。

## 記事フォルダ内の主なファイル

記事フォルダ内の主なファイルは、原則として次の順で使います。

1. `brief.md`
   - 記事の発注書、設計書、先方ディレクター指示を保存する
2. `research.md`
   - 調査メモ、参考情報の要約、論点整理を残す
3. `outline.md`
   - 見出し構成と記事全体の流れを整理する
   - 初期段階で `## 構成案から読み取れる記事の狙い` を作り、構成案の意図と読後ゴールを箇条書きで残す
4. `post-<管理番号>_stepx-x_<短いタイトル>.md`
   - 記事本体を書く
   - `draft.md` / `final.md` は使わない
   - 例: `post-001_step6-2_信頼できる不動産業者を見抜く10のチェックポイント.md`

補足:

- `log.md` は全工程で使う進行記録ファイルとして、各ステップの途中でも随時更新する
- `log.md` には、実施内容、未対応事項、次の担当への引き継ぎを残す
- 作業終了時は、上記の「作業終了時の作業報告フロー」に沿って `log.md` とチャットに同じ作業報告を残す
