# collabit writing

Collabit の執筆作業領域です。

## Rules

- 元資料そのものは必要に応じて GDrive に置いてよい
- ただし、執筆に必要な要約、観点整理、構成案はこの配下に Markdown で残す
- 記事数が増える前提なので、作業状態ごとにフォルダを分ける
- ステータス順が分かるように、進行フォルダは番号つきで運用する
- 1 記事は 1 フォルダ単位で管理する
- あとで確認したい箇所は、原稿中でピンクマーカーを付けて明示する
- 原稿中のメモは、赤字の `（...）` で記載する

## Structure

```text
writing/
├── README.md                 # この writing 配下の使い方
├── _inbox/                   # 未整理メモ、素材、着手前の断片
├── _assets/                  # 執筆共通で使う軽量な画像、図版、補助資料
├── 1_stock/                  # 今後書く候補、企画ストック、テーマ案
├── 2_in_progress/            # 現在執筆中の記事フォルダ
├── 3_review/                 # 確認待ち、赤入れ待ち、修正待ちの記事フォルダ
├── 4_published/              # 公開済み、納品済みの記事フォルダ
├── 5_archive/                # ボツ案、旧版、役目を終えた記事フォルダ
└── _templates/               # 記事フォルダや原稿用のひな形
```

## Article Directory Unit

各記事は、各ステータスフォルダ配下で 1 記事 1 フォルダで持ちます。

```text
2_in_progress/
└── 260426_articletitle/
    ├── brief.md              # 記事の発注書、設計書、先方ディレクター指示
    ├── research.md           # 調査メモ、参考情報の要約、論点整理
    ├── outline.md            # 見出し構成、記事全体の流れ
    ├── draft.md              # 本文の下書き
    ├── final.md              # 公開版、納品版、確定版の本文
    ├── log.md                # 作業記録、進行メモ、引き継ぎ
    └── assets/               # その記事専用の画像、図版、補助資料
        ├── thumbnail/        # サムネイル、アイキャッチ、カバー画像
        └── description/      # 記事説明用画像、本文図版、補助ビジュアル
```

## Article File Meanings

- `brief.md`
- `research.md`
- `outline.md`
- `draft.md`
- `final.md`
- `log.md`
- `assets/`

### `brief.md`

- 記事の発注書、設計書、前提共有のためのファイル
- 記事の目的、想定読者、検索意図、CTA、納期を書く
- 先方ディレクターからの指示、依頼メモ、注意事項もここに保存する

### `research.md`

- 調査メモ、参考リンク、一次情報の要約を置く
- 元資料が GDrive にある場合も、執筆に必要な要点だけはここに残す

### `outline.md`

- 見出し構成、論点設計、記事全体の流れを書く
- 本文執筆前の設計図として使う

### `draft.md`

- 本文の下書き
- 推敲前の作業版として使う

### `final.md`

- 公開版、納品版、確定版の本文
- 最終的に参照する原稿をここに残す

### `log.md`

- 作業記録、進行メモ、引き継ぎメモを残す
- 誰がどこまで進めたか、次に何をするか、確認待ち事項などを書く

### `assets/`

- その記事専用の画像、図版、補助資料を置く
- 重い元素材や共有前提ファイルは必要に応じて GDrive を使う
- `thumbnail/` はサムネイル、アイキャッチ、SNS 用カバー画像に使う
- `description/` は記事説明用画像、本文図版、補助ビジュアルに使う

## Suggested Flow

- `1_stock/`
  - 企画候補を置く
  - この中の `.md` を指定して「作業開始」と依頼されたら、そのファイルを今回着手する原稿として扱う
- `2_in_progress/`
  - 着手した記事フォルダを置く
  - 着手時は `_templates/yymmdd_template` を複製して記事フォルダを作成し、対象の原稿 `.md` を `1_stock/` からその中へ移動する
- `3_review/`
  - 確認待ちの状態に進める
- `4_published/`
  - 公開済みの状態に移す
- `5_archive/`
  - 廃止、旧版、不要になったものを移す

## Writing Workflow

1. 原稿を受領する
   - ディレクター指示、元原稿、補足資料を確認する
2. 構成案の解釈と前提整理をする
   - 原稿を通読する
   - 見出しと中身の整合性を確認する
   - 粒度のズレ、唐突さ、偏りを確認する
   - この記事で何を解決するのかを整理する
   - 構成の順番と役割を確認する
   - JOB理論目線のアプローチを入れられるか検討する
3. 不明点を洗い出し、先に解消する
   - 気になる点、矛盾しそうな点を洗い出す
   - 他記事との役割分担を確認する
   - 必要に応じて先に質問し、疑問を解消しておく
4. 迷子防止アンカーを置く
   - 指示内容を `brief.md` に書く
   - 記事の軸、やらないこと、注意点を `brief.md` や Notion に残す
5. リサーチ設計をする
   - 何を調べれば書けるかを整理する
   - どこまで調べるか、何を関連記事へ逃がすかを決める
6. リサーチと一次ファクト確認をする
   - 不足情報の調査
   - 数値、制度、表現の確認
   - HowMa機能や関連記事との整合を確認する
7. 本文用図版、説明画像を設計する
   - まず必要箇所を判断し、ラフ設計を行う
   - 図にするべき箇所と文章で済ませる箇所を分ける
8. 本文を設計、執筆する
   - H3本文を書く
   - H3タイトルを整える
   - H2リードを書く
   - H2タイトルを整える
   - まとめを書く
   - 導入リードを書く
   - 導入リードを再確認する
9. 本文用図版、説明画像を仕上げる
   - 本文の確定に合わせて内容を詰める
   - 完成した素材は `assets/description/` に置く
10. 校正する
   - 矛盾
   - 誤字脱字、てにをは
   - 表記ゆれ
   - 重複
   - 論理の流れ
   - トンマナ
   - 読者視点のわかりやすさ
   - CTA接続
11. ファクトチェックを最終実施する
   - 執筆後に事実関係をもう一度確認する
12. サムネイルを制作する
   - 記事内容と訴求がずれないように整える
   - 完成した素材は `assets/thumbnail/` に置く
13. 最終確認をする
   - 指示反映、CTA、トンマナ、納品状態を確認する

## Writing Terms

- `荒書き`
  - 文体を整える前に、とにかく内容を通す段階
- `ラフ`
  - もっと軽く、ざっくりした下書き
- `たたき台`
  - 議論や修正前提のベース案
- `初稿`
  - ひとまず一通り書き切った版
